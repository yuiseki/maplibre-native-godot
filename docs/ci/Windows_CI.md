# Windows CI Notes

Lessons learned from getting the Windows GitHub Actions build working.
This document exists because Windows CI is significantly harder than Linux/macOS
due to vcpkg, MSVC, and maplibre-native's own build system interactions.

## Status: PASSING

The Windows CI build passes as of commit 4266d80. This document captures what we
learned to avoid repeating failed approaches.

---

## maplibre-native uses its own internal vcpkg

maplibre-native ships a vendored vcpkg binary at `platform/windows/vendor/vcpkg/vcpkg.exe`.
During cmake configure, `windows.cmake` runs `Get-VendorPackages.ps1` via
`execute_process()`. That script:

1. Finds a free drive letter (Z:, Y:, X:, ... scanning from Z downward)
2. Mounts `platform/windows/vendor/vcpkg/` as a virtual drive via `subst`
3. Runs `vcpkg.exe install curl dlfcn-win32 libuv libjpeg-turbo libpng libwebp [icu]`
4. Unmounts the virtual drive

The installed packages land at the physical path:
```
maplibre-native/platform/windows/vendor/vcpkg/installed/x64-windows/
```

**This includes ICU.** ICU is installed by default unless `MLN_USE_BUILTIN_ICU` is set.

**Critical consequence:** The runner's vcpkg (`$VCPKG_INSTALLATION_ROOT`) is NOT used
for ICU. ICU ends up in maplibre-native's internal vcpkg installed dir, not in
`C:\vcpkg\installed\` or `build/maplibre-native-windows/vcpkg_installed/`.

## ICU::i18n target must be defined before including MapboxCoreTargets.cmake

`build/maplibre-native-windows/MapboxCoreTargets.cmake` (generated by the
maplibre-native build) references `ICU::i18n`, `ICU::uc`, `ICU::data` as IMPORTED
CMake targets in the `mbgl-core` link interface. CMake validates these targets exist
when the file is included.

**Fix:** pass `maplibre-native/platform/windows/vendor/vcpkg/installed/x64-windows`
as the first entry of `CMAKE_PREFIX_PATH` so `find_package(ICU)` succeeds before
`include("${_mln_targets}")` runs.

`CMakeLists.txt` also has a layered fallback:
1. `find_package(ICU QUIET)` — succeeds when the path is in `CMAKE_PREFIX_PATH`
2. If that fails: look for `icuin.lib` in `MLN_SOURCE_DIR/platform/windows/vendor/vcpkg/installed/x64-windows/` and create INTERFACE targets pointing to those `.lib` files
3. If that also fails: create empty stub INTERFACE targets (cmake configure will succeed but linker will likely fail)

## Cache strategy

| What | Key | Path(s) cached |
|------|-----|----------------|
| maplibre-native build | `mln-windows-<commit-sha>-v6` | `build/maplibre-native-windows`, `maplibre-native/vendor/wgpu-native/target/release/wgpu_native.dll`, `maplibre-native/platform/windows/vendor/vcpkg/installed` |
| godot-cpp build | `godot-cpp-windows-<branch>-Release` | `build/godot-extension-windows/godot-cpp` |

**Key lesson:** The cache must include `maplibre-native/platform/windows/vendor/vcpkg/installed`
(the internal vcpkg installed tree). Previous attempts cached `maplibre-native/vcpkg_installed`
or `build/maplibre-native-windows/vcpkg_installed` — neither exists; packages go to the
internal vcpkg path above.

## CMAKE_TOOLCHAIN_FILE for maplibre-native

We pass `-DCMAKE_TOOLCHAIN_FILE="$env:VCPKG_INSTALLATION_ROOT/scripts/buildsystems/vcpkg.cmake"`
to maplibre-native's cmake configure. This uses the runner's vcpkg for things that
the runner's vcpkg handles, while `Get-VendorPackages.ps1` uses the internal vcpkg
for curl, ICU, dlfcn-win32, libuv, etc.

**Note from old docs:** An earlier version of the docs said "Do NOT pass CMAKE_TOOLCHAIN_FILE
to maplibre-native" — this was based on a different experiment. The current working
approach DOES pass it, and it works.

## GDExtension-only vcpkg dependencies

The GDExtension cmake requires packages that maplibre-native does NOT install via its
internal vcpkg (or that we need to find via the runner's vcpkg toolchain):

```yaml
- name: Install GDExtension vcpkg dependencies
  run: |
    & "$env:VCPKG_INSTALLATION_ROOT\vcpkg.exe" install `
      dlfcn-win32:x64-windows `
      libuv:x64-windows `
      libpng:x64-windows `
      zlib:x64-windows
```

**Do NOT install `icu` here.** maplibre-native compiles `mbgl-core.lib` against ICU from
its internal vcpkg. Installing a different ICU version via the runner's vcpkg causes
LNK2019 (unresolved symbols with `_74` version suffixes) because the two ICU versions
differ.

## CMake 4.x + godot-cpp 4.3 incompatibility

See `docs/build-guide/Windows_11.md` gotcha #3. The same fix applies in CI:

```yaml
"-DCMAKE_CXX_FLAGS_DEBUG=/MDd /Zi /Ob0 /Od /RTC1"
```

This is passed to the GDExtension cmake configure to prevent a CMake 4.x `string(REPLACE)`
error in godot-cpp's CMakeLists.txt when `CMAKE_CXX_FLAGS_DEBUG` is empty.

## Failed approaches (chronological)

1. **Dynamically finding vcpkg_installed after build** — `find build/maplibre-native-windows
   -path "*/vcpkg_installed/x64-windows"` returns nothing because packages don't go there.

2. **Caching `maplibre-native/vcpkg_installed`** — This path never exists. maplibre-native
   has no `vcpkg.json` manifest; packages go to the internal vcpkg, not the runner's vcpkg.

3. **Caching `C:/vcpkg/installed/x64-windows`** — The runner's vcpkg never installs ICU
   (we never ask it to). Not the right path.

4. **Installing `icu:x64-windows` via runner's vcpkg** — Installs ICU 76, but `mbgl-core.lib`
   was compiled against ICU 74. LNK2019 with symbols like `ucnv_open_74`.

5. **PowerShell em-dash in Write-Host string** — `Write-Host "Z:\ exists — searching..."` caused
   `TerminatorExpectedAtEndOfString`. Replace `—` with `-` and add `continue-on-error: true`.
